# 게임 대회 팀원 경매 시스템 설계

## 1) 전제 조건
- 대회 운영진은 게임 운영진과 별개이며, 게임 내부 API/로그/시스템에 접근하지 못함.
- 따라서 참가자 정보는 **신청서 + 운영진 검증 데이터 + 이전 대회 DB** 중심으로 운영.

## 2) 가격 산정 정책

### A. AI 연결 가능 시
1. 연결 가능한 AI 공급자(OpenAI/Anthropic/Google)를 탐지.
2. 후보 가격 산정 프롬프트 입력(플레이어 프로필, 포지션 수요/공급, 이전 대회 낙찰가 등).
3. 다중 AI 연결 시 각 AI의 산정 근거를 비교/요약(“토론 로그”).
4. 운영진이 최종 시작가 확정.

### B. AI 연결 불가 시(필수 알고리즘)
입력 데이터(우선순위): 사용자 이름, 최고 티어, 포지션, 선호 캐릭터

알고리즘:
1. 티어 기준 내림차순 정렬
2. 정렬 상태에서 포지션별 분류
3. 포지션별 인원 수를 운영진에게 제시
4. 운영진 입력
   - 포지션 보정값(0.5 ~ 1.0)
   - 티어별 기본가격(문자 티어만 허용: 예: 다이아, 실버)
5. 희소성 계산
   - `희소성 = Demand / Supply`
6. 최종 가격
   - `가격 = 티어기본가격 × 포지션보정값 × 희소성계수 × 랜덤계수(0.9~1.1)`
7. 운영진 최종 확정

## 3) 복수 게임 지원 아키텍처
- `game_profiles` 테이블로 게임별 규칙 분리
  - 포지션 체계(예: 탱커/딜러/서포터, 탑/정글/미드/원딜/서폿)
  - 팀별 필요 포지션 수(Demand 계산에 사용)
  - 티어 표준화 맵(게임별 티어 표현 차이 보정)
- 공통 경매 엔진 + 게임별 설정(플러그인/설정 파일) 구조

## 4) 관람/재미 요소
- 실시간 경매 대시보드
  - 현재 매물, 현재가, 남은 시간, 입찰 기록
- 통계 패널
  - 포지션별 평균 낙찰가, 팀 예산 잔여, 티어별 낙찰가 분포
- 이벤트 요소
  - 랜덤 스폰서 보너스(예산 소폭 보정)
  - 와일드카드 라운드(특정 라운드 추가 슬롯)
  - 커뮤니티 예측(관람자 투표)


## 5) 예산 정책
- 운영진은 대회 생성 시 `team_budget_cap`을 일괄 지정.
- 팀은 해당 예산 내에서만 입찰 가능.
- 입찰 시 서버에서 잔여 예산 검증 후 수락/거절 처리.

## 6) 데이터 저장 및 이력 활용
모든 데이터 영구 저장:
- 플레이어 프로필/신청서
- 시작가 산정 근거(AI 토론 또는 알고리즘 입력값)
- 입찰 이력(누가, 얼마, 언제)
- 낙찰 결과
- 시즌/대회 메타데이터


이전 대회 데이터 활용:
- 동일 티어/포지션 낙찰가 평균 및 분산
- 특정 플레이어 과거 낙찰 기록
- AI 가격 산정 시 참고 피처로 제공

## 7) 운영 플로우
1. 대회 생성(게임 선택, 팀 수, 팀 예산, 포지션 수요 정의)
2. 참가자 등록(티어/포지션/선호 캐릭터)
3. 시작가 산정(AI or 알고리즘)
4. 운영진 검토 및 확정


## 8) 검증 규칙(중요)
- 티어 입력 시 숫자 포함 금지 (`다이아1` 불가, `다이아` 허용)
- 포지션 보정값 범위 제한(0.5 ~ 1.0)
- 랜덤계수 범위 제한(0.9 ~ 1.1)
- 공급(Supply) 0일 때 예외 처리(경고 후 수동가격)
